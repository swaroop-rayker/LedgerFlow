# ADR-005: Clean Architecture

## Context & Problem Statement
LedgerFlow must remain highly maintainable, extensible, and clean as it grows. The business rules must never depend on UI or Framework modules.

## Proposed Options
1.  **Monolithic MVVM Package Layout**: A single module where separation is done via folders.
2.  **Clean Architecture Modules**: Separate modules (Core, Domain, Data, Presentation, Services, App) enforcing strict compile dependency structures.

## Decision Outcome
Selected Option: **Clean Architecture Modules**

### Justification
*   **Compile-Time Verification**: Prevents developers from mistakenly referencing Room entities in UI screens or ViewModels in repository classes.
*   **Extensibility**: Adding future modules (like cloud sync or plugin engines) only requires implementing defined domain repository interfaces.
*   **Test Isolation**: Allows domain logic to compile and test rapidly using pure Kotlin JVM parameters.

## Consequences
*   **Pros**: Compile-level protection against architectural rot, independent feature modules, easy separation of concerns.
*   **Cons**: Increases Gradle build configuration scripts and Hilt DI boilerplate declarations.
