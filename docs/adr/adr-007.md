# ADR-007: Hierarchical Categories

## Context & Problem Statement
Financial categories require subcategory groupings (e.g. "Food" -> "Groceries", "Restaurants"). Creating distinct database tables for Categories and Subcategories increases join query logic and entity mapping complexity.

## Proposed Options
1.  **Separate Category and Subcategory Tables**: High schema normalization.
2.  **Tag-Only Classification**: No formal hierarchy; transactions are categorized using arbitrary tags.
3.  **Self-Referencing Hierarchical Category Table**: A single `categories` table with a nullable self-referencing `parent_id` column.

## Decision Outcome
Selected Option: **Self-Referencing Category Table**

### Justification
*   **Unified Schema**: Eliminates duplicate logic. A subcategory is just a Category entity that has a non-null `parent_id`.
*   **Simple Hierarchy**: Standard tree structures are constructed by grouping rows by `parent_id`.
*   **Flexible Scaling**: Accommodates future depth configurations (e.g. multi-tier hierarchies) without database migrations.

## Consequences
*   **Pros**: Simplified schema, zero duplicated code, easy UI rendering tree models.
*   **Cons**: Requires compile-level recursive cycle protection to prevent infinite loops in UI trees.
