# ADR-003: SQLCipher Database Encryption

## Context & Problem Statement
Because all financial data is stored locally, losing the device or having it compromised presents a security risk. The database files (SQLite) must be securely encrypted at rest.

## Proposed Options
1.  **Standard Unencrypted SQLite/Room**: Zero encryption; database is readable by root-level utilities.
2.  **App-Level Field Encryption**: Selective column encryption (encrypting strings manually before saving).
3.  **SQLCipher full page-level encryption**: Transparent page-by-page database encryption.

## Decision Outcome
Selected Option: **SQLCipher Transparent Encryption**

### Justification
*   **Full Coverage**: Encrypts everything—indexes, metadata, cross-references, and rows.
*   **Performance**: High-speed, page-level encryption running on natively compiled binaries.
*   **Seamless Room Integration**: Room natively supports custom SQLite open helper factories, allowing SQLCipher to drop in with minimal code modification.

## Consequences
*   **Pros**: Complete data protection at rest, compatible with Room, minor performance overhead.
*   **Cons**: Slightly increases APK size (~3-5MB for SQLCipher native libraries), complicates background migrations (as keys must be resolved first).
