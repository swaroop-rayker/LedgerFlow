# ADR-006: Two-Tier Pending SMS Database Queue

## Context & Problem Statement
A biometric-authenticated Keystore key cannot decrypt a database key while the device is locked. If an incoming banking SMS is intercepted in the background while the screen is off, writing directly to the main secure database will fail.

## Proposed Options
1.  **Lower Main DB Security**: Store the main DB key using boot-accessible keystore wrappers (no biometrics required).
2.  **Ignore Background SMS**: Parse SMS only when the app is active and unlocked.
3.  **Two-Tier Encryption Queue**: Write to a temporary queue DB encrypted with boot-accessible keys, and merge into the main biometric-secured DB when the app is unlocked.

## Decision Outcome
Selected Option: **Two-Tier Encryption Queue**

### Justification
*   **Keeps Security High**: The primary ledger remains encrypted behind biometric verification at all times.
*   **Enable Background Automation**: Background workers can write to the queue immediately without prompting the user.
*   **User Verification**: The merge interface allows the user to categorize or reject incoming transactions before they are written to the main ledger.

## Consequences
*   **Pros**: Automatic SMS parsing is supported while locked, main DB security is not compromised.
*   **Cons**: Requires managing two separate database open helpers and a merge interface.
