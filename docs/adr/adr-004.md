# ADR-004: Repository Pattern

## Context & Problem Statement
The business layer (domain use cases) must remain unaffected by data layer schema modifications, file changes, or local preferences structures.

## Proposed Options
1.  **Direct DAO Injection**: Inject Room DAOs directly into ViewModels or Use Cases.
2.  **Repository Pattern**: Map DAOs to domain interface boundaries.

## Decision Outcome
Selected Option: **Repository Pattern**

### Justification
*   **Decoupling**: Domain models are kept separate from Room entity annotations.
*   **Testing Flexibility**: Allows mock implementations (e.g. mock repositories) in unit tests without SQLCipher/Room database frameworks.
*   **Source Abstraction**: A repository can merge Room database data with Jetpack DataStore preferences seamlessly.

## Consequences
*   **Pros**: Modular separation, easy mocking, data layer source changes don't touch business use cases.
*   **Cons**: Introduces mapping overhead (converting `Entity` objects to clean `Domain` objects).
